Several fixes needed across the patient EHR Treatment tab and session completion flow:
1. All services from the invoice must create separate Treatment history entries. When a session is completed with multiple billable services (e.g. Tooth Filling + Teeth Cleaning), every one must write its own treatment record — not just one. Check the "Complete Session & Queue Invoice" submit handler: confirm it loops over all service/procedure line items and creates a record for each (a common bug is using Supabase .insert() with a single object instead of an array when multiple items should be inserted). Confirm tooth number, date, dentist, clinic branch, and clinical notes are attributed correctly per-service, not all defaulting to the same values.
2. Remove the "FOLLOW-UP" tag/label entirely from Treatment tab history cards. This is redundant — follow-up scheduling already has its own dedicated Follow-ups tab. Remove the tag/badge rendering from each Treatment card without altering the underlying follow-up data itself (date, scheduled status) — that data must still correctly power the Follow-ups tab.
3. Clinical notes show "–" instead of actual content. Notes typed in "Add Treatment Record" (or via the session invoice) aren't persisting/displaying in Treatment history. Check whether notes are being saved on submit and whether the history view reads the correct field.
4. Timestamps for the same event don't match across views. E.g. a session/audit log shows one time (1:17 PM) while the corresponding Treatment record shows a different time (12:27 PM) for what should be the same recorded moment. Confirm whether both are reading from the same database column, or generated independently (client-side new Date() vs. server-side timestamp, or a timezone conversion bug). We already have a shared date.ts utility — audit all timestamp displays (session logs, Treatment tab, invoice, follow-ups) and ensure they all route through it consistently instead of inline Date logic, sourced from one canonical database value.
5. Tooth number logic must be consistent. Tooth number should be null/hidden for whole-mouth services (Teeth Cleaning, Consultation, Braces) and populated only for tooth-specific ones (Tooth Filling, Tooth Extraction). Define which services require a tooth number in one shared config/constant (single source of truth) used by the Treatment form, invoice, and treatment history — not scattered conditionals. Fix any logic causing whole-mouth services to incorrectly carry a stale tooth number (e.g. bleeding over from a Dental Chart tab selection).
Check Supabase schema (treatment records, session/audit log tables) before implementing to confirm correct columns and insert shapes.
